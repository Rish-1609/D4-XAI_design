Build this as one connected Inventory Traceability System inside D4.

1. What the system should be

One unified system where:

materials are created in master data
locations are defined in a warehouse hierarchy
pallets/cartons/items get an identity
barcode and RFID both map to that identity
every scan is validated against business rules
valid scans post inventory movements
stock balances and traceability history update automatically

Do not build RFID as a separate standalone module.

3. Correct logic flow

The system should always work like this:

create object → assign barcode/RFID → scan/input → resolve object → validate → post movement → update stock → save history

Not:
scan → show event on dashboard

4. How identities should work
Barcode

D4 should generate barcode values.

RFID

RFID tag EPC should be assigned and mapped in D4.

Printer

Barcode printers should be supported because you will need labels for:

inbound relabeling
internal pallet/carton creation
production output
repacking
damaged label reprint

Rule:

D4 creates the identity
printer prints the label
RFID maps to the same identity

Best practice:
Use handling unit ID as the main barcode key.

Example:

Handling Unit = PALT-00045
Material = RM001
Batch = BT-2024-001
Barcode = PALT-00045
RFID EPC = 02180C000625
5. Barcode vs RFID logic
Barcode

Use for operator-confirmed actions:

receiving
putaway
transfer
issue

RFID

Use for automated detection:

dock entry/exit
zone movement

location detection
unauthorized movement alerts
Combined rule
barcode = confirmation
RFID = automated visibility
both update the same inventory engine

8. Required validations

Before any movement is posted, validate:

scanned object exists
barcode/tag is active
quantity is available
batch/serial is valid
location is correct
destination is allowed
linked document is correct
stock is not on hold
no duplicate movement is being posted

If invalid, create an exception. Do not update stock.

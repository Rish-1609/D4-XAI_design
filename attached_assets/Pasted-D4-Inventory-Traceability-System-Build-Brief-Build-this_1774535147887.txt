D4 Inventory Traceability System Build Brief

Build this as a working inventory traceability system, not a standalone RFID dashboard.

1. Goal

Create a unified Barcode + RFID Inventory Tracking system inside the Inventory module that supports:

inbound receipt
putaway
internal transfers
issue to production
production receipt
outbound / dispatch
QC hold / release
cycle count / stock adjustment
full material traceability

RFID and barcode must both feed the same inventory logic.

2. Core principle

Do not build RFID as a separate standalone feature.

Build one common engine:

barcode scan
RFID scan
manual entry
API event

All of these should create or validate the same inventory movements.

6. Barcode logic

Barcode scanning should be used for operator-confirmed transactions.

When barcode is scanned:

read barcode value
resolve to item / pallet / batch / material
check current transaction context
validate location, quantity, status, and document
post movement if valid
log exception if invalid

Barcode should support:

receiving
putaway
transfer
issue
pack
dispatch
count
hold / release
7. RFID logic

RFID should be used as an automated event source, not as the only truth.

When RFID event is received:

capture raw read
resolve RFID EPC to mapped object
resolve reader to location
infer event type based on rules
either:
create informational event
propose movement
auto-post confirmed movement if rules allow
log exception if mapping or rules fail

RFID should support:

dock entry / exit
zone visibility
passive movement detection
bulk movement confirmation
rapid stock locating
loss prevention / unauthorized movement alerts
8. Barcode + RFID combined rule

Use this model:

barcode = transactional confirmation
RFID = automated visibility and detection

Example:

barcode confirms receipt or issue
RFID confirms movement through dock / zone

Both must update the same inventory records.

9. Inbound flow

Build inbound like this:

user opens receive screen
selects PO / ASN
scans barcode or RFID-tagged handling unit
system matches material, batch, qty to PO line
creates stock_in movement
inventory status becomes received or QC pending
current location = inbound dock / receiving area
user can then perform putaway to rack/bin

If supplier labels are missing:

allow D4 to create internal barcode / handling unit
allow RFID tag assignment
10. Putaway flow
user selects received unit
scans source pallet / carton / item
scans destination rack / bin
system validates destination
posts putaway / internal transfer
updates current location
11. Internal transfer flow
user selects transfer action
scans source inventory object
scans destination location
validates stock exists and is movable
posts internal_transfer
updates inventory by location
12. Issue to production flow
user opens production issue
selects production order
scans pallet / carton / item
validates material and batch against production order
posts issue_to_production
deducts stock from warehouse location
updates traceability history
13. Production receipt flow
user opens production receipt
selects production order
creates finished goods batch / lot
creates barcode / handling unit
optionally assigns RFID tag
posts production_receipt
updates finished goods inventory
14. Outbound / dispatch flow
user opens shipment / dispatch
selects shipment order
scans pallet / carton / item
validates right material, batch, qty, location
posts stock_out
if RFID reader detects exit at dock, log confirmation
if tag seen at outbound dock without shipment, raise exception
15. QC hold / release flow

Support:

hold stock
release stock

Rules:

held stock cannot be issued or shipped
hold / release must be written to inventory ledger
history must remain visible
16. Cycle count / adjustment flow
user opens count session
scans barcode or RFID items in zone
system compares expected vs actual
creates adjustment proposal
approved adjustment posts to ledger
keep audit trail
17. Required validation rules

Before posting any movement, validate:

item exists
tag / barcode exists
tag active
correct location
correct document
enough quantity
status allows movement
batch / serial valid
no duplicate transaction
destination allowed

If validation fails, do not post stock movement. Create exception instead.

21. Event processing design

Use 3 layers:

Layer 1: raw capture

Store every incoming barcode / RFID event in scan_events_raw

Layer 2: resolution

Resolve:

what was scanned
which location
which user / device
which transaction
valid or invalid
Layer 3: posting

If valid:

create movement event
update inventory balance
update current location
write ledger
write audit trail

If invalid:

write exception event
